# Bitácora Técnica Detallada - Proyecto Crea PDF Puppeteer

Esta bitácora extiende el registro de cambios general, profundizando en los archivos modificados, las funciones específicas implementadas y la lógica técnica detrás de cada versión.

---

## [v1.9.2] - 12 de Marzo de 2026
**Objetivo**: Mejorar la auditoría y facilitar la recuperación de documentos ya procesados.

### 1. Descarga Directa de PDFs (Historial)
- **Archivos Modificados**:
  - `src/services/datosFormatoService.ts`: Se modificaron las funciones `listDatosFormato` y `searchDatosFormatoByReporte`.
  - `src/views/js/app.js`: Lógica de renderizado de lista en `loadDraftsList`.
- **Detalle Técnico**:
  - **Backend**: Se agregó `include: { pdfsGenerados: { take: 1, orderBy: { fechaCreacion: 'desc' } } }` a las consultas Prisma. Esto permite traer la ruta del archivo físico (`archivoPath`) asociado al borrador sin hacer una segunda consulta.
  - **Frontend**: Se detecta si el objeto `d.pdfsGenerados` tiene elementos. Si es así, se inyecta un botón HTML `<button class="btn-download-pdf">` con un atributo `data-url`.
  - **Manejo de Eventos**: Se implementó `ev.stopPropagation()` en el evento click del botón para evitar que, al querer descargar, se abra involuntariamente el formulario de edición.

### 2. Auditoría de Usuarios (Visibilidad Admin)
- **Archivos Modificados**:
  - `src/services/datosFormatoService.ts`: Se agregó `include: { usuario: true }` en las consultas.
  - `src/views/js/app.js`: Se implementó lógica condicional `if (currentUser.role === 'ADMIN')`.
- **Detalle Técnico**:
  - El backend ahora expone el objeto `usuario` (nombre, email) en el JSON de respuesta de `/api/drafts`.
  - El frontend compara el rol de la sesión actual. Si es ADMIN, inyecta un `div` con el nombre y correo del creador en color verde oscuro para diferenciación visual.

## [v1.9.1] - 10 de Marzo de 2026
**Objetivo**: Endurecer las reglas de negocio para el registro de nuevos usuarios.

### 1. Validación de Dominio Institucional
- **Archivos Modificados**: `src/app.ts` (Endpoint `/api/register`).
- **Lógica**:
  - Se implementó una validación estricta usando `String.prototype.endsWith('@conagua.gob.mx')` antes de procesar cualquier dato.
  - Rechazo inmediato con código HTTP 400 si el dominio no coincide.

### 2. Protección de Integridad y Seguridad
- **Archivos Modificados**: `src/app.ts`.
- **Lógica**:
  - **Bloqueo Admin**: `if (email === 'admin@conagua.gob.mx') return 403;`. Evita la suplantación de la cuenta maestra mediante el formulario público.
  - **Anti-Duplicidad**: Consulta previa `findUserByEmail(email)` antes de invocar `prisma.usuario.create`. Esto previene errores de restricción única (Unique Constraint) de SQL a nivel de aplicación, permitiendo devolver mensajes de error amigables al usuario ("El correo ya existe") en lugar de errores crudos de base de datos.

## [v1.9.0] - 09 de Marzo de 2026
**Objetivo**: Mejorar la navegabilidad y la experiencia de registro (UX).

### 1. Gestión de Sesión en UI
- **Archivos Modificados**: 
  - `src/app.ts`: Nuevo endpoint `/api/me`.
  - `src/views/index.html`: Nuevo contenedor `#session-info`.
  - `src/views/js/app.js`: Lógica de arranque (`DOMContentLoaded`).
- **Detalle Técnico**:
  - El endpoint `/api/me` verifica `req.session.userId`. Si existe, devuelve `{ userId, userName, role }`. Si no, devuelve 401.
  - El frontend consume este endpoint al cargar. Si recibe 401, hace un `window.location.href = '/login'`.
  - Botón Logout consume `/api/logout`, el cual ejecuta `req.session.destroy()` en el servidor.

### 2. Visibilidad de Contraseña (Toggle)
- **Archivos Modificados**: `src/views/login.html`.
- **Detalle Técnico**:
  - **Librería**: Integración de *Bootstrap Icons* vía CDN.
  - **Lógica JS**: Función `togglePassword(inputId, btn)` que alterna el atributo `type` del input entre `"password"` y `"text"`.
  - **UX**: Se fuerza `input.focus()` después del click para mantener el teclado activo en dispositivos móviles y evitar que la animación CSS "cierre" el formulario.

## [v1.8.0] - 06 de Marzo de 2026
**Objetivo**: Transformación a Arquitectura Multiusuario Segura.

### 1. Sistema de Autenticación
- **Tecnologías**: `express-session` (Manejo de estado), `bcryptjs` (Hashing), `cookie-parser`.
- **Archivos Clave**:
  - `src/services/userService.ts`: Abstracción de Prisma para `createUser` y `findUserByEmail`.
  - `src/app.ts`: Middleware `requireAuth` que intercepta todo tráfico a `/api/*` y `/templates/*`.
- **Base de Datos**: 
  - Nueva tabla `model Usuario { id, email, password, nombre, organismo, direccion }`.
  - Relación `1:N` entre `Usuario` y `DatosFormato`.

### 2. Lógica de Privacidad (Row Level Security Aplicada)
- **Archivos Modificados**: `src/services/datosFormatoService.ts`.
- **Detalle Técnico**:
  - Las funciones de listado (`listDatosFormato`, `listPendientes`) ahora aceptan un parámetro `usuarioId`.
  - **Lógica condicional**: `const whereClause = (usuarioId && userType !== 'ADMIN') ? { usuarioId } : {};`.
  - Esto garantiza que Prisma añada `WHERE usuario_id = X` automáticamente en todas las consultas, aislando la información por usuario a menos que sea Admin.

### 1. Flujo "Solicitar Reporte" (Client-Side)

  - El historial filtra visualmente estos registros con una etiqueta de advertencia roja.

## [v1.6.0] - [v1.6.1] - Dockerización
**Objetivo**: Despliegue agnóstico del entorno.

### 1. Dockerfile Optimizado
- **Imagen Base**: `node:20-slim` (Debian ligero).
- **Reto Técnico**: Puppeteer requiere librerías gráficas de Linux (Chromium dependencies).
- **Solución**: Se instalaron paquetes `apt-get` específicos (`libnss3`, `libatk1.0`, etc.) y se configuró Puppeteer para usar el Chrome instalado en el contenedor en lugar de descargar uno local.

### 2. Docker Compose
- **Servicios**: `app` (Node.js) y `db` (PostgreSQL de Alpine).
- **Volúmenes**: Persistencia de datos en `./pgdata`.
- **Networking**: Red interna `backend-network` para comunicación segura entre API y DB.

## [v1.0.0] - [v1.5.0] - Fases Iniciales
- **Motor de Renderizado**: Uso de `puppeteer` para inyectar HTML sobre imágenes JPG de alta resolución (300 DPI) simulando los formatos oficiales.
- **Sistema de Plantillas**: Archivos `.html` con placeholders `{{VARIABLE}}` que se reemplazan en tiempo de ejecución (Regex).
- **Manejo de CSV**: Uso de `papaparse` en frontend para cargar catálogos masivos (Claves presupuestales, Áreas) sin sobrecargar el servidor.
